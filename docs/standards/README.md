# Standards

This section contains the project standards that govern product, engineering, and writing conventions.

## Merge Rule

No code is merged unless the documentation explaining why it exists is merged with it.

## Required Documentation

For any substantive change, the accompanying documentation should explain:

- why the change exists
- what assumptions it encodes
- what trade-offs were accepted
- what evidence supports it

## GitHub Practice

- Use issues for work tracking.
- Use pull requests for review and merge.
- Use ADRs for durable decisions.
- Keep `main` protected after initialization ends.
